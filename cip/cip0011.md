## Summary

A correspondence check between `main-version.ipynb` (the original monolithic notebook) and the numbered `NN_*.ipynb` series revealed 10 substantive analysis or visualisation cells that are either absent or only stub-present in the numbered notebooks. This CIP plans how to restore each missing cell — expressed via appropriate `access` / `assess` / `address` library calls — and identifies the small number of new library functions required.

The mapping artefact is recorded in `notebook_cell_mapping.json` (repo root).

## Motivation

CIP-0010 refactored the monolithic notebook into a structured series (`00_` through `05_`). However, the correspondence check shows that several analyses present in `main-version.ipynb` did not survive the migration:

- Two cross-technology heatmaps (concerns and benefits)
- The data-quality summary chart
- Volume / top-cluster comparison cells
- The full robustness section (stable core, lexical novelty), which are stubs in `05_robustness.ipynb`
- The traceability evidence-pack export cells

Without these, the numbered notebooks are not a faithful replacement for `main-version.ipynb` and the paper workflow is incomplete.

## Detailed Description

### Gap inventory (from `notebook_cell_mapping.json`)

All items below are `no_match` (similarity < 70 %) against every NN notebook.

#### Group A — Data quality (`00_data_quality.ipynb`)

| main cell | title | library coverage |
|-----------|-------|-----------------|
| main[11] | Summarise paragraph-level data quality | `assess.plot_data_quality()` exists |

The function `assess.plot_data_quality()` (or `assess.plot_quality()`) was introduced in CIP-0010. The notebook just needs a thin cell that calls it.

#### Group B — Cross-technology heatmaps (`02_shared_structure.ipynb`)

| main cell | title | library coverage |
|-----------|-------|-----------------|
| main[40] | Visualise cross-technology concern distribution | **No library function yet** |
| main[77] | Visualise cross-technology benefit distribution | **No library function yet** |

These are heatmaps of the top-N concern/benefit clusters against all technologies. The data already lives in `concern_salience_df` / `benefit_salience_df` (outputs of `address.concern_salience()` / `address.benefit_salience()`). A new `address.cross_technology_heatmap(salience_df, cluster_labels, kind)` function is needed to encapsulate the plotting logic and keep inline code out of the notebook.

The cells belong in `02_shared_structure.ipynb`, after the dendrogram cells and before the embedding-space visualisation.

#### Group C — Volume and top-cluster summaries (`02_shared_structure.ipynb`)

| main cell | title | library coverage |
|-----------|-------|-----------------|
| main[87] | Compare volume (phrase counts and paragraph incidence) | `address._volume_table()` exists (private) |
| main[88] | Top clusters side-by-side (overall) | `address._top_clusters()` exists (private) |

`02_shared_structure.ipynb` already imports these via a comment stub. The fix is to make them public (`address.volume_table()`, `address.top_clusters()`) and add output cells that display the tables.

#### Group D — Robustness (`05_robustness.ipynb`)

| main cell | title | library coverage |
|-----------|-------|-----------------|
| main[101] | Assess robustness of stable core and AI fingerprint (concerns) | `address.ai_fingerprint_over_crosscut()` exists; stable-core bootstrap **not yet** |
| main[111] | Assess robustness of stable core and AI fingerprint (benefits) | Same |
| main[103] | Assess lexical novelty in AI concerns over time | **No library function yet** |

`05_robustness.ipynb` has placeholder comment cells for these but no working code. The stable-core robustness check (`ai_fingerprint_over_crosscut` over bootstrap samples) needs a thin public wrapper. The lexical novelty analysis (year-on-year Jaccard similarity of concern phrase vocabulary) needs a new `address.lexical_novelty_over_time(df, kind)` function.

#### Group E — Evidence-pack exports (`05_robustness.ipynb`)

| main cell | title | library coverage |
|-----------|-------|-----------------|

The HTML evidence pack and xlsx exports exist as inline logic in `main-version.ipynb`. A new public `address.export_evidence_pack(df, kind, output_folder)` function is needed to encapsulate this, and corresponding cells added to `05_robustness.ipynb`.

### Library changes required (all in `pub_dialogue/address.py`)

| Function | Status | Notes |
|----------|--------|-------|
| `address.cross_technology_heatmap(salience_df, cluster_labels, kind)` | **New** | Heatmap of top-N clusters × technologies |
| `address.volume_table(concerns_df, benefits_df)` | **Rename from `_volume_table`** | Make public |
| `address.top_clusters(df, summary_df, kind, n=10)` | **Rename from `_top_clusters`** | Make public |
| `address.stable_core_robustness(df, embeddings, kind, n_trials=20)` | **New** | Bootstrap AI fingerprint stability |
| `address.lexical_novelty_over_time(df, kind)` | **New** | Year-on-year Jaccard novelty of phrase vocab |
| `address.export_evidence_pack(df, cluster_summary_df, kind, output_folder)` | **New** | HTML + xlsx traceability export |

`assess.plot_data_quality()` is already public — no change needed there.

### Fynesse phase assignment

Every new/restored cell must respect the access → assess → address boundary:

- **`00_data_quality`** (`assess` phase): call `assess.plot_data_quality(chunks_df)`. No address calls.
- **`02_shared_structure`** (`address` phase): call `address.cross_technology_heatmap(...)`, `address.volume_table(...)`, `address.top_clusters(...)`. Inputs are already-computed artefacts loaded from disk.
- **`05_robustness`** (`address` phase): call `address.stable_core_robustness(...)`, `address.lexical_novelty_over_time(...)`, `address.export_evidence_pack(...)`. All receive pre-computed artefacts; no raw data access.

None of the new cells should call `access.data()` or read raw PDFs — that remains in `01_processing.ipynb`.

## Implementation Plan

1. **Library: make `_volume_table` and `_top_clusters` public**
   - Rename to `volume_table` and `top_clusters` in `address.py`
   - Update any internal callers
   - Add unit tests

2. **Library: add `cross_technology_heatmap()`**
   - Input: `salience_df` (docs × clusters), `cluster_labels` dict, `kind` ("concern"|"benefit"), optional `top_n=40`
   - Output: a `plotly` figure (consistent with other visualisations in the project)
   - Unit test with synthetic salience matrix

3. **Library: add `stable_core_robustness()`**
   - Bootstrap: repeatedly sample 80 % of documents, recompute AI fingerprint, report Jaccard stability of top-20 clusters
   - Returns a summary DataFrame and prints a text report
   - Unit test with small synthetic data

4. **Library: add `lexical_novelty_over_time()`**
   - For each consecutive pair of years, compute Jaccard similarity of the set of concern/benefit phrase tokens
   - Returns a DataFrame; the notebook cell then plots it
   - Unit test with small synthetic phrase list

5. **Library: add `export_evidence_pack()`**
   - Encapsulates the HTML paragraph-level evidence export and xlsx traceability table from main-version cells 93 and 109
   - Input: `df` (phrases), `cluster_summary_df`, `kind`, `output_folder`
   - Unit test: check files are written with expected columns

6. **Notebook: `00_data_quality.ipynb`** — restore Group A cell
   - Add cell after existing quality cells: `assess.plot_data_quality(chunks_df)`

7. **Notebook: `02_shared_structure.ipynb`** — restore Group B and C cells
   - Add two cross-technology heatmap cells (concern then benefit) after dendrogram cells
   - Add volume-comparison cell and top-clusters side-by-side cell (replace stubs)

8. **Notebook: `05_robustness.ipynb`** — restore Group D and E cells
   - Replace stub comment cells with working cells for stable-core robustness (concern + benefit)
   - Replace stub comment cells with working cells for lexical novelty (concern + benefit)
   - Add export evidence-pack cells (concern + benefit)

9. **Update `notebook_cell_mapping.json`**
   - Re-run the mapping script after all cells are added; confirm no-match count drops to ≤ infrastructure-only cells (setup, Colab file upload, etc.)

## Backward Compatibility

- Renaming `_volume_table` → `volume_table` and `_top_clusters` → `top_clusters` is a public API addition. Because these were previously private (underscore-prefixed), no user code should break.
- No changes to checkpoint file formats or existing public API.

## Testing Strategy

- Unit tests for each new library function added to `tests/test_address.py`
- Run all numbered notebooks end-to-end on the standard checkpoint set and confirm outputs are produced
- Re-run `notebook_cell_mapping.json` generator and verify remaining `no_match` cells are limited to infrastructure cells (install, Colab upload, config) that legitimately differ
