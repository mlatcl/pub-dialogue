# CIP-0006: Fix chunk extraction filter and cache integrity

## Status

- [x] Proposed - Initial idea documented
- [ ] Accepted - Approved, ready to start work
- [ ] In Progress - Actively being implemented
- [ ] Implemented - Work complete, awaiting verification
- [ ] Closed - Verified and complete
- [ ] Rejected - Will not be implemented
- [ ] Deferred - Postponed

## Summary

Two related corpus integrity bugs that compound each other. First, the
chunk-extraction filter (`extract_chunks_from_pdf`) has a Python indentation
bug that means the MIN_CHUNK_WORDS/MIN_CHUNK_CHARS filter has no effect — every
paragraph including table-of-contents lines, page footers, and headings is
kept, producing ~63% sub-50-word chunks in the 12,528-paragraph corpus. Second,
the extraction cache is validated only by matching chunk ID sets; a cache
produced by a partially-failed run (where transient API errors silently returned
empty) is indistinguishable from a complete run, so re-runs silently reuse
bad data. Together these mean the paper's "12,528 paragraphs" and "15%
paragraph yield" figures are both wrong.

## Motivation

From the review:

> `chunks.append()` is unindented, so every paragraph is kept regardless of
> length. The MIN filter has no effect. 12,528 total chunks, 63% below the
> supposed 50-word minimum, 29% below 10 words.

> The cache validity check only verifies that the set of chunk IDs matches, not
> that all of them have non-failed entries. A cache from a partial run looks
> identical to a complete run.

**Paper impact:**
- "12,528 paragraphs processed" is technically the count but misleading — most
  are noise fragments, not substantive paragraphs.
- "~15% of paragraphs yielded a concern phrase" is actually 7.2% (899/12,528);
  the 14.5% figure confuses "concern phrases per chunk" with "chunks yielding ≥1
  concern".
- 41 of 66 documents produced zero concerns — partially attributable to the
  cache masking failed extraction runs.

## Detailed Description

### Fix 1: Chunk extraction filter

Move `chunks.append(...)` inside the `if word_count >= min_chunk_words` block
so the filter actually applies. Also:
- Add a `meets_min_filter: bool` column to allow post-hoc analysis of excluded
  chunks (useful for verifying the filter isn't too aggressive).
- Add a `word_count` column (already partially present) for transparency.
- Report the filtering statistics at the end of the extraction cell:
  total paragraphs found, kept, discarded.

**Decision required:** Does the analysis include or exclude sub-50-word chunks?
Options:
- A) Apply filter strictly — reduces corpus to ~4,500–5,000 substantive chunks.
  Paper Methods section must be updated with the new numbers.
- B) Keep all chunks but track them — `meets_min_filter` column, extraction
  skips short chunks (returns sentinel), paper reports both counts.

Recommendation: Option A. The sub-50-word fragments are mostly structural noise
and the LLM reliably returns NO_CONCERN/NO_BENEFIT for them anyway.

### Fix 2: Cache integrity

After loading a cache, validate that it is complete and not a partial-failure
cache. A cache entry counts as a failure if:
- `concerns` (or `benefits`) is an empty list **and** the chunk meets the
  minimum filter — i.e. it is substantive but yielded nothing.

The fix: add a `_failed` key to each cache entry set to True when the
extraction returned an error rather than a genuine sentinel. On cache load,
count the number of `_failed` entries. If `_failed` > 0 and the user hasn't
explicitly opted to reuse the cache, warn and prompt to re-extract.

Alternatively, store `ExtractionResult` objects (from CIP-0001) directly in
the cache so error vs. sentinel is always distinguishable.

### Fix 3: Re-examine silent documents

After fixing the cache, the 41 documents that produced zero concerns need to be
re-diagnosed:
1. Run extraction on just those 41 files with full logging (from CIP-0001).
2. Check the extraction_errors.csv — were they API failures or genuine empties?
3. If API failures: re-extract and recheck all downstream analysis.

## Backward Compatibility

- The chunk schema gains two new columns (`word_count`, `meets_min_filter`); no
  existing downstream code reads these by name.
- The cache schema gains a `_failed` key per entry; old caches without this key
  are treated as "not failed" (safe default).
- Applying the filter will reduce the corpus size and change all downstream
  counts. The paper's Methods section **must** be updated. This is expected.

## Testing Strategy

- Unit test: verify `extract_chunks_from_pdf` excludes chunks below min_words
- Unit test: verify chunk output includes `word_count` and `meets_min_filter`
- Unit test: cache loader warns when `_failed` entries present
- Integration check: recount chunks from one known PDF and verify against manual count
