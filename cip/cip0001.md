# CIP-0001: Extraction yield diagnostics and filter review

## Status

- [x] Proposed - Initial idea documented
- [x] Accepted - Approved, ready to start work
- [x] In Progress - Actively being implemented
- [x] Implemented - Work complete, awaiting verification
- [x] Closed - Verified and complete
- [ ] Rejected - Will not be implemented
- [ ] Deferred - Postponed

## Summary

Add diagnostic instrumentation to the concern and benefit extraction stages of the pipeline so that empty extraction results are classified into three categories: design-intent empties (LLM returned the sentinel value), tech-word filter drops (the LLM produced a phrase that was then removed by the substring filter), and silent errors (an API exception was caught and swallowed). This implements [REQ-0001](../requirements/req0001_extraction-yield-diagnostics.md).

## Motivation

Jess's observation that "so many paragraphs are not yielding concerns or benefits" is a valid concern, but currently it is impossible to answer with data. The pipeline conflates three distinct causes:

1. The LLM correctly returns `NO_CONCERN` / `NO_BENEFIT` for administrative or process text (expected and correct behaviour)
2. The LLM produces a phrase, but the substring filter drops it because it contains a term from `tech_words` (e.g. `"ai"` matching mid-word in `"tailoring"`, or any phrase about `"artificial intelligence"` that still contains the word)
3. An API exception occurs, is caught silently by a bare `except:` clause, and the paragraph returns an empty list — indistinguishable from case 1

Without distinguishing these, it is not possible to know whether the extraction yield is behaving as designed or whether bugs and over-filtering are artificially depressing results. This also feeds into [CIP-0004](./cip0004.md) (artefact reduction), since understanding the filter's behaviour is a prerequisite for revising it.

## Detailed Description

### Current extraction structure

Both `extract_concerns_from_paragraph` and `extract_benefits_from_paragraph` follow this pattern:

```python
try:
    response = client.chat.completions.create(...)
    lines = response.choices[0].message.content.strip().split('\n')
    phrases = [l.strip() for l in lines if l.strip() and l.strip() != 'NO_CONCERN']
    # tech-word filter
    phrases = [p for p in phrases if not any(t in p.lower() for t in tech_words)]
    return phrases
except:
    return []
```

The problem is that `return []` appears in both the "nothing found" branch (after LLM returns sentinel) and the `except:` branch (after any failure). The tech-word filter also silently removes phrases between these two steps without any record.

### Proposed changes

**Step 1: Structured return type**

Change the extraction functions to return a structured result instead of a bare list:

```python
from dataclasses import dataclass, field
from typing import List, Optional

@dataclass
class ExtractionResult:
    chunk_id: str
    raw_phrases: List[str]          # LLM output before filter
    retained_phrases: List[str]     # after tech-word filter
    dropped_by_filter: List[str]    # phrases removed by tech-word filter
    sentinel_returned: bool         # True if LLM returned NO_CONCERN/NO_BENEFIT
    error: Optional[str] = None     # exception message if one occurred
```

**Step 2: Specific exception handling**

Replace `except:` with `except Exception as e:` and store `str(e)` in the result, rather than silently returning empty.

**Step 3: Post-extraction summary**

After all paragraphs are processed, compute and write `extraction_yield_summary.csv`:

| column | description |
|---|---|
| `track` | `concern` or `benefit` |
| `total_chunks` | total chunks sent for extraction |
| `sentinel_empties` | chunks where LLM returned sentinel |
| `filter_drops_chunks` | chunks where ≥1 phrase was dropped by tech filter |
| `filter_drops_total` | total phrases dropped by tech filter across all chunks |
| `error_chunks` | chunks where an exception occurred |
| `retained_phrases` | total phrases retained |

**Step 4: Filter drop log**

Write `tech_filter_drops.csv` with columns: `chunk_id`, `dropped_phrase`, `matching_tech_word`. This allows assessment of whether the filter is over-aggressive (e.g. dropping valid phrases via substring matches like `"ai"` in `"tailoring"`).

**Step 5: Error log**

Write `extraction_errors.log` (or `.csv`) with columns: `chunk_id`, `track`, `exception_type`, `exception_message`.

### Scope boundary

This CIP does **not** propose changing the tech-word filter logic, changing the prompt, or changing the cluster count. Those are addressed in [CIP-0004](./cip0004.md) and [CIP-0003](./cip0003.md). The goal here is only to make the existing behaviour observable.

## Implementation Plan

1. **Define `ExtractionResult` dataclass** in the notebook setup cell (or a shared utility cell near the top)
   - Fields: `chunk_id`, `raw_phrases`, `retained_phrases`, `dropped_by_filter`, `sentinel_returned`, `error`

2. **Refactor `extract_concerns_from_paragraph`**
   - Return `ExtractionResult` instead of list
   - Use `except Exception as e:` instead of bare `except:`
   - Record phrases before and after filter

3. **Refactor `extract_benefits_from_paragraph`** — same changes

4. **Update the extraction loop** for both concerns and benefits
   - Collect `ExtractionResult` objects
   - After loop, build `extracted_concerns.csv` / `extracted_benefits.csv` as before (from `retained_phrases`)
   - Additionally write `extraction_yield_summary.csv`, `tech_filter_drops.csv`, `extraction_errors.log`

5. **Add a summary print** at the end of the extraction cell showing the yield breakdown

6. **Test on a small sample** (e.g. 50 chunks) before running on the full corpus

## Backward Compatibility

- `extracted_concerns.csv` and `extracted_benefits.csv` schemas are unchanged (they are built from `retained_phrases` as before)
- Three new output files are added but do not affect downstream cells
- Checkpoint logic (`save_checkpoint` / `load_checkpoint`) should be updated to store `ExtractionResult` objects rather than bare lists, but the checkpoint file names can remain the same

## Testing Strategy

- Run extraction on a known sample of 20–50 chunks where the expected output is known (e.g. from a previous manual review)
- Verify that `extraction_yield_summary.csv` sums correctly (total = sentinel_empties + error_chunks + chunks_with_retained_phrases)
- Deliberately introduce a bad chunk ID to trigger an exception and verify it appears in `extraction_errors.log`
- Check `tech_filter_drops.csv` for any obviously valid phrases that were incorrectly dropped
