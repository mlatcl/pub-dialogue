# CIP-0012: Externalise LLM Prompts to Editable YAML Files

## Status

- [x] Proposed - Initial idea documented
- [x] Accepted - Approved, ready to start work
- [ ] In Progress - Actively being implemented
- [ ] Implemented - Work complete, awaiting verification
- [ ] Closed - Verified and complete
- [ ] Rejected - Will not be implemented
- [ ] Deferred - Postponed

## Summary

LLM prompts are currently hardcoded as Python string constants in `pub_dialogue/address.py`. This CIP moves the stable, template-style prompts (those with a single `{text}` placeholder) into an editable YAML file, making them inspectable and tuneable by domain researchers without touching Python code. Dynamic prompts built from runtime values (cluster summaries, exemplar phrases, etc.) remain in Python but have their fixed sub-strings extracted where practical.

## Motivation

`address.py` currently contains:

- **Six extraction prompt constants** (`EXTRACTION_PROMPT`, `BENEFIT_EXTRACTION_PROMPT`, and the B/C paraphrase and minimal variants) — pure text templates with a `{text}` placeholder. These are research artefacts that a qualitative researcher would want to adjust without modifying source code.
- **Four dynamic prompts** (framing-lens, retry, `label_cluster`, `label_benefit_cluster`) — assembled at runtime from f-strings that embed cluster summaries, exemplar phrases, and `kind`. Fully externalising these requires a templating engine and brings minimal benefit.
- **Two named variant dicts** (`CONCERN_PROMPT_VARIANTS`, `BENEFIT_PROMPT_VARIANTS`) that are already treated as experimental variables by `run_prompt_sensitivity()` (CIP-0008). Their natural representation is a keyed YAML structure.

Keeping prompts inside Python has several costs:
- A domain expert wishing to refine a prompt must navigate `address.py`, understand Python string literal syntax, and re-run tests — unnecessary friction.
- A Git diff mixing a prompt wording change with code changes obscures which kind of change was made.
- The sensitivity-analysis variant dicts are duplicated between Python constants and the dict — a YAML file gives a single source of truth.

### YAML file layout

A new file `pub_dialogue/prompts.yml` (committed, shareable):

```yaml
# LLM prompt templates for pub_dialogue.
# All extraction prompts use {text} as the paragraph placeholder.
# Dynamic prompts (lens, retry, label) are assembled in address.py;
# only their static system messages are stored here.

extraction:
  concern:
    default: A_current          # which variant is used by extract_phrases()
    A_current: |
      Extract the core public concerns from this paragraph.
      ...
      Paragraph:
      {text}
    B_paraphrase: |
      Identify the specific anxieties, objections, or reservations ...
      ...
      {text}
    C_minimal: |
      List the public concerns in this paragraph. ...
      {text}
  benefit:
    default: A_current
    A_current: |
      Extract the core public BENEFITS ...
      {text}
    B_paraphrase: |
      Identify the hoped-for gains ...
      {text}
    C_minimal: |
      List the public benefits expressed in this paragraph. ...
      {text}

system_messages:
  qualitative_researcher: "Expert qualitative researcher. Return only valid JSON."
  engagement_analyst: "Expert in public engagement and discourse analysis. Return only valid JSON."
```

The file is committed to the repository. Local experimentation is done by editing the file directly, or by passing an explicit `prompts` dict to `run_prompt_sensitivity()` — no separate override file is needed.

### Loading mechanism

A new function `pub_dialogue.address.load_prompts(path=None)` loads the prompt YAML:

```python
def load_prompts(path: Optional[Path] = None) -> dict:
    """Load prompt templates from YAML.

    Loads pub_dialogue/prompts.yml by default.
    Pass an explicit path to load a different file (useful in tests).
    """
```

`EXTRACTION_PROMPT`, `BENEFIT_EXTRACTION_PROMPT`, `CONCERN_PROMPT_VARIANTS`, and `BENEFIT_PROMPT_VARIANTS` become module-level constants derived from `load_prompts()` at import time, so all existing call sites continue to work unchanged:

```python
_PROMPTS = load_prompts()
EXTRACTION_PROMPT = _PROMPTS["extraction"]["concern"]["A_current"]
BENEFIT_EXTRACTION_PROMPT = _PROMPTS["extraction"]["benefit"]["A_current"]
CONCERN_PROMPT_VARIANTS = {
    k: v for k, v in _PROMPTS["extraction"]["concern"].items()
    if k not in ("default",)
}
BENEFIT_PROMPT_VARIANTS = {
    k: v for k, v in _PROMPTS["extraction"]["benefit"].items()
    if k not in ("default",)
}
```

System message strings in the dynamic labelling/lens functions are replaced by lookups into `_PROMPTS["system_messages"]`.

### Failure mode

If `prompts.yml` is missing, `load_prompts()` falls back to a minimal set of hardcoded defaults (the current Python constant values) and emits a `logging.warning`. This means the library remains usable in environments where the YAML file has not been installed alongside the package.

### Backward compatibility

- All existing public constants (`EXTRACTION_PROMPT`, `BENEFIT_EXTRACTION_PROMPT`, `CONCERN_PROMPT_VARIANTS`, `BENEFIT_PROMPT_VARIANTS`) remain available at the same names with the same values.
- `extract_phrases()` and `run_prompt_sensitivity()` signatures are unchanged.
- The `default:` key in the YAML controls which variant `extract_phrases()` uses; changing it in the file alters the default without any code change.

## Implementation Plan

1. **Create `pub_dialogue/prompts.yml`** with the full text of all six extraction prompts and the four system messages. Use YAML block scalars (`|`) to preserve whitespace.

2. **Add `load_prompts()`** to `address.py` (or a new `pub_dialogue/config.py`). Use `PyYAML`. Implement the merge with `machine.yml` if present.

3. **Replace Python string constants** with values derived from `load_prompts()` at module load. Keep the same names so no call site changes.

4. **Replace system message literals** in `label_cluster`, `label_benefit_cluster`, and the lens/retry prompt blocks with lookups into `_PROMPTS["system_messages"]`.

5. **Tests**: add `tests/test_prompts.py`:
   - Loading from the default path returns the expected prompt text.
   - `EXTRACTION_PROMPT` and `BENEFIT_EXTRACTION_PROMPT` match the YAML values.
   - `CONCERN_PROMPT_VARIANTS` and `BENEFIT_PROMPT_VARIANTS` have the correct keys.
   - `load_prompts(path=<missing file>)` falls back to defaults without raising.

## Backward Compatibility

No breaking changes. All existing public API (constants, function signatures) is preserved. The only observable behaviour change: if a user edits `prompts.yml`, the new wording takes effect on the next Python import.

## Testing Strategy

- Unit tests in `tests/test_prompts.py` (see Implementation Plan step 6).
- Existing `tests/test_address.py` extraction tests continue to pass unchanged (they call `extract_phrases()` which uses the same constants).
- Sensitivity analysis tests in `test_address.py` continue to pass; `run_prompt_sensitivity()` still accepts an explicit `prompts` dict that overrides the YAML defaults.
