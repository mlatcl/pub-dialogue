## Summary

Create a documented, repeatable process for validating the dialogue analyser's results using the traceability outputs (`traceability_paragraphs.csv`, `cluster_exemplars.json`, `evidence_pack_paragraphs.html`). The playbook should be usable by a researcher without programming knowledge and should address Jess's questions about how to go through results in depth and how to check extracted paragraphs against the export pack. Implements [REQ-0005](../requirements/req0005_results-validation.md).

## Motivation

The pipeline produces several traceability outputs designed to support result checking, but there is no guide for how to use them. Without a documented process:

- It is unclear which clusters to check first and how many to check
- There are no criteria for deciding that a cluster is suspicious and needs investigation
- It is not obvious how to link an extracted phrase in `extracted_concerns.csv` back to the original PDF paragraph
- Jess and Neil may spend time checking the same things, or miss important checks

This CIP produces a validation playbook that answers: *When you have the export pack, what do you do with it?*

## Detailed Description

### Available traceability outputs

| File | Contents |
|---|---|
| `traceability_paragraphs.csv` | Links each extracted phrase to its source chunk (chunk_id, source document, paragraph text) |
| `cluster_exemplars.json` | For each cluster: the cluster ID, label, and the 5–10 most central extracted phrases |
| `cluster_labels.json` | Cluster IDs with human-readable labels and framing lens assignments |
| `cluster_summary.csv` | Cluster sizes, document coverage, cross-cutting flag |
| `evidence_pack_paragraphs.html` | HTML view: for each cluster, the exemplar phrases and the source paragraphs they came from |
| `extracted_concerns.csv` | All retained concern phrases with chunk_id provenance |
| `extracted_benefits.csv` | All retained benefit phrases with chunk_id provenance |
| `paragraph_chunks.csv` | Source chunk text and metadata (document name, year, technology) |

### Playbook structure

The playbook will be a markdown document at `validation_playbook.md` in the project root (or `docs/validation_playbook.md`). It covers four validation activities:

---

**Activity 1: Cluster coherence spot-check**

*Goal:* Confirm that the top clusters are semantically coherent — the label describes the exemplars, and the exemplars describe a genuine concern or benefit.

*Steps:*
1. Open `cluster_summary.csv` in a spreadsheet. Sort by cluster size (descending).
2. Take the top 10 clusters by size.
3. For each cluster, open `evidence_pack_paragraphs.html` and navigate to that cluster.
4. Read the label, the exemplar phrases, and 3–5 source paragraphs.
5. Ask: Does the label accurately describe the exemplar phrases? Do the source paragraphs contain the concern/benefit described?
6. Flag the cluster if: the label does not match the exemplars, the exemplars are generic process language (see REQ-0004), or all exemplars come from a single document.
7. Repeat with 5 randomly selected clusters (use a random number generator or the 5 smallest cross-cutting clusters).

*Record findings in:* a simple spreadsheet with columns: cluster_id, label, coherent (Y/N/Partial), notes.

---

**Activity 2: Cross-cutting claim check**

*Goal:* Verify that clusters marked as cross-cutting (appearing across multiple technologies) genuinely do so.

*Steps:*
1. In `cluster_summary.csv`, filter to rows where `cross_cutting = True`.
2. For each of the top 10 cross-cutting clusters, open `traceability_paragraphs.csv` and filter to that cluster_id.
3. Check the `technology` or `document_name` column: are there at least 3 distinct technologies represented?
4. Flag any cross-cutting cluster that is actually dominated (>60% of phrases) by a single technology.

---

**Activity 3: Source paragraph verification**

*Goal:* Confirm that extracted phrases are faithful representations of the source paragraph's content.

*Steps:*
1. In `extracted_concerns.csv` (or `extracted_benefits.csv`), select 20 rows at random.
2. For each row, find the `chunk_id` and look up the source paragraph text in `paragraph_chunks.csv` (match on `chunk_id`).
3. Read the source paragraph and the extracted phrase side by side.
4. Ask: Is the concern/benefit phrase a fair, decontextualised representation of what the paragraph expresses?
5. Flag any phrase that: (a) does not appear to relate to the paragraph text, (b) contains vocabulary from the extraction prompt (see REQ-0004), or (c) is so generic it could apply to any document.

*Record flagged chunk_ids and the reason.*

---

**Activity 4: Export pack completeness check**

*Goal:* Confirm the export pack contains the expected files and that counts are consistent.

*Steps:*
1. Unzip the export pack.
2. Verify the following files are present: `paragraph_chunks.csv`, `extracted_concerns.csv`, `extracted_benefits.csv`, `cluster_summary.csv`, `cluster_exemplars.json`, `cluster_labels.json`, `traceability_paragraphs.csv`, `evidence_pack_paragraphs.html`.
3. Check record counts:
   - `paragraph_chunks.csv` row count should match the total chunks reported at extraction
   - `extracted_concerns.csv` + sentinel_empties + filter_drops + errors should equal total chunks (use `extraction_yield_summary.csv` from CIP-0001)
   - `cluster_summary.csv` should have exactly `N_CONCERN_CLUSTERS` rows for concerns
4. Flag any discrepancies.

---

### Suspicious cluster criteria (summary)

A cluster should be flagged for further investigation if any of the following are true:

- The cluster label does not match the exemplar phrases
- > 60% of phrases in the cluster come from a single document
- The cluster appears in only one technology's documents
- The exemplar phrases contain primarily meta-vocabulary ("public dialogue", "engagement", etc.)
- The cluster has fewer than 3 source documents contributing phrases

### Format

The playbook will be a standalone markdown file. It will not require Python to use — all steps work with a spreadsheet (CSV) and a web browser (HTML). Where Python is useful for filtering, short optional code snippets will be provided in collapsible blocks.

## Backward Compatibility

- The playbook is a new document; it does not change any existing output files
- The optional addition to the export cell adds the playbook to the zip, which is an additive change
- The "validation summary" addition to `evidence_pack_paragraphs.html` adds content to an existing output; it does not break existing structure
