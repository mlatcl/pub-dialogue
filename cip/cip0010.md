# CIP-0010: Stage Classes and Config Centralisation

## Status

- [x] Proposed - Initial idea documented
- [x] Accepted - Approved, ready to start work
- [ ] In Progress - Actively being implemented
- [x] Implemented - Work complete, awaiting verification
- [x] Closed - Verified and complete
- [ ] Rejected - Will not be implemented
- [ ] Deferred - Postponed

## Summary

Replace the current mix of module-level functions and notebook-level constants with three
**stage classes** ‚Äî `AccessStage`, `AssessStage`, `AddressStage` ‚Äî that own all pipeline
configuration as typed dataclass properties and expose pipeline operations as methods.
Notebooks become readable narratives: they instantiate a stage object and call its methods
rather than redeclaring the same constants in every cell.

This builds on CIP-000D (access/assess/address structure) and CIP-000F (notebook import
clean-up) and completes the transition to a coherent, testable library beneath the notebooks.

## Motivation

Five analysis notebooks each independently define the same set of constants:

| Constant | Value | Notebooks |
|---|---|---|
| `N_CONCERN_CLUSTERS` | 75 | 01a, 02, 03, 04, 05 |
| `N_BENEFIT_CLUSTERS` | 75 | 01a, 02, 03, 04, 05 |
| `RANDOM_SEED` | 42 | 01a, 05 |
| `TECH_COL` | `"technology_meta"` | 02, 03, 04, 05 |
| `AI_TECH_LABEL` | `"AI"` | 03, 04, 05 |
| `CROSS_CUTTING_ENTROPY_THRESHOLD` | 0.5 | 01a, 02, 03 |
| `SOFT_MEMBERSHIP_THRESHOLD` | 0.3 | 01a |
| `VALIDATION_SAMPLE_N` | 250 | 05 |
| `OUTPUT_FOLDER` / `CHECKPOINT_FOLDER` | `Path("outputs")` etc. | all |

In addition, several analysis cells contain non-trivial computation that was inserted as a
fix for missing notebook cells but logically belongs in the library:

- `ai_year` / `benefit_ai_year` ‚Äî year √ó cluster count matrices (04, 05)
- `traj` / `benefit_traj` ‚Äî PCA trajectory DataFrames (04)
- `salience_df` / `benefit_salience_df` ‚Äî per-technology cluster profiles (03)

Centralising these eliminates silent inconsistency: changing `N_CONCERN_CLUSTERS` currently
requires editing five files.

## Detailed Description

### Design: three `@dataclass` stage classes

Each stage module gains a single dataclass whose fields are typed configuration values with
defaults matching the existing hardcoded values. All existing module-level functions are
kept unchanged; the dataclass methods delegate to them.

```python
# pub_dialogue/access.py
from dataclasses import dataclass, field
from pathlib import Path

@dataclass
class AccessStage:
    output_folder: Path = Path("outputs")
    checkpoint_folder: Path = Path("checkpoints")
    pdf_folder: Path = Path("pdfs")
    min_chunk_words: int = MIN_CHUNK_WORDS      # module constant kept as default
    max_chunk_words: int = MAX_CHUNK_WORDS
    min_chunk_chars: int = MIN_CHUNK_CHARS

    def load_artifacts(self) -> dict:
        """Delegate to existing load_artifacts(output_folder, checkpoint_folder)."""
        return load_artifacts(self.output_folder, self.checkpoint_folder)

    def extract_chunks_from_pdf(self, pdf_path: Path) -> pd.DataFrame:
        return extract_chunks_from_pdf(pdf_path, ...)
```

```python
# pub_dialogue/assess.py
@dataclass
class AssessStage:
    access: AccessStage

    def validate_extraction_cache(self, ...): ...
    def plot_data_quality(self, ...): ...
    def generate_validation_summary(self, ...): ...
```

```python
# pub_dialogue/address.py
@dataclass
class AddressStage:
    access: AccessStage
    n_concern_clusters: int = 75
    n_benefit_clusters: int = 75
    random_seed: int = 42
    tech_col: str = "technology_meta"
    ai_tech_label: str = "AI"
    cross_cutting_threshold: float = CROSSCUTTING_ENTROPY_THRESHOLD  # 0.5
    soft_membership_threshold: float = 0.3
    validation_sample_n: int = 250

    # --- computation methods (lifted from notebook cells) ---
    def concern_year_matrix(self, technology: str | None = None) -> pd.DataFrame:
        """Year √ó concern-cluster count matrix, optionally filtered to one technology."""
        ...

    def benefit_year_matrix(self, technology: str | None = None) -> pd.DataFrame:
        ...

    def concern_trajectory(self, technology: str | None = None) -> pd.DataFrame:
        """PCA 2-D trajectory of mean concern embeddings per year."""
        ...

    def benefit_trajectory(self, technology: str | None = None) -> pd.DataFrame:
        ...

    def concern_salience(self) -> pd.DataFrame:
        """Per-technology √ó concern-cluster relative salience DataFrame."""
        ...

    def benefit_salience(self) -> pd.DataFrame:
        ...

    # --- pipeline methods (phase 3) ---
    def cluster(self, concerns_df, benefits_df, embeddings, ...): ...
    def label_clusters(self, ...): ...
    def assign_framing_lenses(self, client, ...): ...
```

### Notebook pattern after refactor

**Setup cell (all analysis notebooks):**

```python
# @title Load pre-computed artifacts
from pathlib import Path
import pub_dialogue.utils as du
from pub_dialogue.utils import show_status, show_complete
from pub_dialogue.access import AccessStage
from pub_dialogue.address import AddressStage

access  = AccessStage()           # all defaults; override for non-standard paths
address = AddressStage(access=access)
a = access.load_artifacts()

concerns_df = a["concerns_df"]
benefits_df = a["benefits_df"]
# ... other unpacking as before
```

**Analysis cell (example ‚Äî 04_temporal_dynamics):**

Before:
```python
# Compute AI concern year √ó cluster matrix (ai_year)
TECH_COL = 'technology_meta'
ai_concerns_yr = concerns_df[concerns_df[TECH_COL] == 'AI'].copy()
ai_concerns_yr['year'] = ai_concerns_yr['year'].astype(int)
ai_year = pd.crosstab(ai_concerns_yr['year'], ai_concerns_yr['cluster_id'].astype(int))
...
```

After:
```python
ai_year = address.concern_year_matrix(technology=address.ai_tech_label)
```

### Relationship to existing CIPs

- **CIP-000D** established the access/assess/address module split ‚Äî this CIP adds the
  class layer on top of it.
- **CIP-000F** cleaned up notebook imports ‚Äî this CIP further reduces setup-cell boilerplate
  and removes the inserted computation cells.

### What is NOT changed

- All existing module-level functions (`load_artifacts`, `extract_phrases`, `label_cluster`,
  etc.) are kept as-is to preserve backward compatibility with any external callers.
- `utils.py` re-exports remain intact.
- `public_dialogue_analyser_v19.ipynb` is out of scope (it is the legacy monolith).

## Implementation Plan

The refactor is broken into four independently-deliverable phases. Each phase ends with all
existing tests passing and all five analysis notebooks executing clean via `nbconvert`.

> **Sequencing constraint**: CIP-0010 must be implemented *after* CIP-0009 (temporal
> methodology revision). CIP-0009 changes the temporal notebook cells that Phase 2 of this
> CIP lifts into `AddressStage` methods. Implementing Phase 2 before CIP-0009 is merged
> would create conflicts requiring re-work. CIP-0008 (prompt sensitivity) can proceed in
> parallel since it touches different notebooks.

### Phase 1 ‚Äî Config centralisation (lowest risk)

Add the three dataclasses as **config holders only** (no new computation methods yet).
Update all notebook setup cells to instantiate them and read config from them.

- Add `AccessStage` to `pub_dialogue/access.py`
- Add `AssessStage` to `pub_dialogue/assess.py`
- Add `AddressStage` to `pub_dialogue/address.py` with config fields only
- Re-export all three from `pub_dialogue/utils.py` and `pub_dialogue/__init__.py`
- Update setup cells in `01a`, `02`, `03`, `04`, `05`
- Replace inline `TECH_COL`, `RANDOM_SEED`, `N_*_CLUSTERS`, `AI_TECH_LABEL`, etc. with
  `address.tech_col`, `address.random_seed`, etc.
- Add `TestAccessStageDefaults`, `TestAddressStageDefaults` tests checking defaults

### Phase 2 ‚Äî Computation methods

Lift the inserted notebook computation cells into `AddressStage` methods:

- `concern_year_matrix()` / `benefit_year_matrix()` ‚Äî crosstab logic currently in 04 & 05
- `concern_trajectory()` / `benefit_trajectory()` ‚Äî PCA logic currently in 04
- `concern_salience()` / `benefit_salience()` ‚Äî salience logic currently in 03
- Remove those cells from notebooks; replace with single-line method calls
- Add unit tests for each method using fixture DataFrames

### Phase 3 ‚Äî Pipeline methods (clustering and labelling)

Wrap the heavier pipeline steps from `01a_clustering.ipynb` into `AddressStage` methods:

- `cluster(concerns_df, embeddings)` ‚Üí runs k-means, returns updated DataFrame + centroids
- `label_clusters(client, centroids, concerns_df)` ‚Üí returns `cluster_labels` dict
- `assign_framing_lenses(client, cluster_labels, concerns_df)` ‚Üí returns
  `framing_lens_mappings`
- Thin out `01a_clustering.ipynb` cells accordingly
- Add tests with mocked `LLMClient`

### Phase 4 ‚Äî AssessStage methods + final notebook cleanup

- Wrap `assess.py` helpers into `AssessStage` methods
- Audit remaining notebook cells for any residual naked constants; remove them
- Update `cip/README.md` to add CIP-0010
- Close CIP-000F (its implementation is now complete)

## Backward Compatibility

All existing module-level functions are kept unchanged. The stage classes are **additive**.
Code that currently does `du.load_artifacts(OUTPUT_FOLDER, CHECKPOINT_FOLDER)` continues to
work; the class is an alternative entry point, not a replacement.

## Testing Strategy

- After each phase: `pytest tests/ -v` must be fully green.
- After each phase: `jupyter nbconvert --to notebook --execute` for all five notebooks.
- New tests follow the existing pattern: pytest classes in `tests/`, `unittest.mock` for LLM
  calls, `tmp_path` for filesystem fixtures.
- Key new test cases:
  - Stage class defaults match the values that were hardcoded in notebooks
  - `concern_year_matrix` / `benefit_year_matrix` return correct shapes
  - `concern_trajectory` returns a DataFrame with `year`, `pc1`, `pc2` columns
  - `cluster()` produces the expected number of clusters
