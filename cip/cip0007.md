# CIP-0007: Remove technology metadata from cluster labelling

## Status

- [x] Proposed - Initial idea documented
- [x] Accepted - Approved, ready to start work
- [x] In Progress - Actively being implemented
- [x] Implemented - Work complete, awaiting verification

- [ ] Rejected - Will not be implemented

## Summary

The `label_cluster` function passes each exemplar phrase to the LLM with its
source technology appended: `"- [phrase] (from AI)"`. This contaminates cluster
labels. A cluster whose concerns happen to come from multiple AI documents will
receive a tech-flavoured label ("AI governance…") even when the actual content
is about something generic. The fix is straightforward — remove the technology
annotation from the exemplar list — but has downstream consequences: all
cluster labels must be regenerated and checked for changes.

## Motivation

From the review:

> In `label_cluster` the exemplars are formatted as `"- {phrase} (from {technology})"`.
> This means the LLM almost certainly incorporates the technology name into the
> label. A cluster of "public concern about data" will be labelled "data concerns
> in AI/genome editing/…" rather than just "data concerns". Since the paper
> presents cluster labels as bottom-up summaries of participant concerns, this
> is a significant methodological problem.

The impact is that the cluster labels may systematically misrepresent the
thematic structure of concerns as technology-specific when they are actually
cross-cutting. This is particularly important for the paper's central claim
about technology-agnostic vs technology-specific concerns.

## Detailed Description

### The fix

Change line ~550 of `dialogue_utils.py`:

```python
# Before:
exemplar_texts = "\n".join(
    f"- {ex[phrase_key]} (from {ex.get('technology', 'Unknown')})"
    for ex in exemplars[:8]
)

# After:
exemplar_texts = "\n".join(
    f"- {ex[phrase_key]}"
    for ex in exemplars[:8]
)
```

### Downstream consequences

After this fix, all cluster labels must be regenerated. The resulting labels
should be reviewed and compared against the current set. Key questions:
1. Do any labels change materially?
2. Do cross-cutting themes become more visible?
3. Does this change the paper's analysis of technology-specific vs. cross-cutting
   concerns?

### Optional enhancement

Consider adding a **technology distribution summary** to the cluster
characterisation step (not to the LLM prompt, but to the notebook output):
for each cluster, report what fraction of its members come from each technology.
This is useful for the analysis but must not bias the labelling.

## Implementation Plan

1. Remove `(from {technology})` from exemplar formatting in `label_cluster`
2. Regenerate all cluster labels (concern clusters and benefit clusters)
3. Compare old vs new labels; document material changes
4. Update paper text where cluster names or characterisations have changed
5. Optionally: add post-hoc technology distribution summary per cluster

## Backward Compatibility

- All cluster label strings will potentially change. Any hardcoded cluster name
  references in the notebook or paper must be reviewed.
- The cache for cluster labels should be invalidated / regenerated.

## Testing Strategy

- Unit test: verify the exemplar list passed to the LLM does not contain
  technology annotations
- Spot-check: for 3–5 clusters, compare old vs. new label strings
