# CIP-0003: Extract dialogue_utils.py, build test suite, strip notebook outputs, align section numbering

## Summary

Extract all shared helper functions from the notebook into a standalone Python module `dialogue_utils.py`, build an associated `pytest` test suite in `tests/`, strip stored cell outputs from the `.ipynb` file, and align section numbering. The notebook becomes a thin orchestration layer that imports from `dialogue_utils`. This is the foundational code quality improvement that makes all other CIPs easier to implement and verify. Implements [REQ-0003](../requirements/req0003_code-auditability.md).

## Motivation

Jess described "getting into a mess on function names" as a concrete problem. The notebook has three interrelated issues that a separate utility module resolves:

**1. Duplicate function definitions.** `label_cluster`, `run_for_k`, `normalized_entropy`, `parse_year`, and several robustness helpers are defined in multiple cells. In a live kernel the last-executed cell wins — effective behaviour depends on execution order, not on reading the source. A separate `.py` file enforces a single authoritative definition.

**2. Stored outputs inflate the file.** The `.ipynb` file is approximately 7 MB, mostly stored cell output JSON. This makes `git diff` nearly unreadable and `git blame` on source lines unusable.

**3. No tests.** With functions duplicated across cells there is no safe way to refactor. A `pytest` suite provides a regression harness that makes subsequent changes (CIP-0001, CIP-0002, CIP-0004) safe to implement.

**4. Inconsistent section numbering.** Part I uses numerals (1–8), Part II uses alphanumerics (2B–8B), Part IV uses Roman/alpha (IV.A, IV.B) — navigable but unnecessarily inconsistent.

### Module structure: `dialogue_utils.py`

```
dialogue_utils.py
├── # --- I/O & checkpointing ---
│   save_checkpoint(), load_checkpoint()
│   show_status(), show_complete(), show_warning()
│
├── # --- Corpus ingestion ---
│   extract_chunks_from_pdf(pdf_path, metadata) -> pd.DataFrame
│
├── # --- Extraction ---
│   ExtractionResult (dataclass)
│   extract_phrases(row_tuple, kind, client, tech_words) -> ExtractionResult
│       (unified replacement for extract_concerns_from_paragraph /
│        extract_benefits_from_paragraph; kind="concern"|"benefit")
│
├── # --- Embeddings ---
│   get_embeddings_batch(texts, model) -> np.ndarray
│
├── # --- Utilities ---
│   parse_year(s) -> int | None
│   tokenize(text) -> list[str]
│   normalized_entropy(counts) -> float
│   hhi(counts) -> float
│
├── # --- Cluster semantics ---
│   label_cluster(cluster_id, exemplars, is_cross_cutting, kind, client) -> str
│   pretty_label(label) -> str
│   clusters_to_labels(df, ...) -> dict
│   clusters_to_lenses(df, ...) -> dict
│
├── # --- Metrics ---
│   ai_fingerprint_over_crosscut(df, kind) -> pd.DataFrame
│   entropy_by_year(df, kind) -> pd.DataFrame
│   topk_share(counts, k) -> float

├── # --- Sensitivity ---
│   run_sensitivity(k, kind, embeddings, df, output_folder, ...) -> None

│
└── # --- Formatting helpers ---
    _volume_table(...), _top_clusters(...), html_escape(s)
```

### Colab access pattern

In Colab the `.py` file needs to be available. Two options:

**Option A (recommended for shared Colab links):** Upload `dialogue_utils.py` as part of the input zip alongside the PDFs. The existing upload cell can be extended to unzip and write `dialogue_utils.py` to `/content/`.

**Option B (for local/repo use):** Mount Google Drive or clone the repo, then `sys.path.insert(0, '/content/repo')`.

The notebook's SECTION 0 setup cell becomes:

```python
# After uploading/mounting the repo
import sys
sys.path.insert(0, '/content')  # or appropriate path
import dialogue_utils as du
```

All subsequent cells call `du.label_cluster(...)`, `du.run_sensitivity(...)`, etc.

### Test suite: `tests/test_dialogue_utils.py`

Tests are grouped by module section. Key test categories:

**Pure utility functions** (fast, no API calls):

```python
def test_normalized_entropy_uniform():
    counts = [10, 10, 10, 10]
    assert abs(du.normalized_entropy(counts) - 1.0) < 1e-6

def test_normalized_entropy_concentrated():
    counts = [100, 0, 0, 0]
    assert du.normalized_entropy(counts) == 0.0

def test_hhi_monopoly():
    assert du.hhi([1, 0, 0]) == pytest.approx(1.0)

def test_parse_year_valid():
    assert du.parse_year("2019") == 2019

def test_parse_year_invalid():
    assert du.parse_year("n/a") is None
```

**ExtractionResult** (from CIP-0001):

```python
def test_extraction_result_filter_tracking():
    result = ExtractionResult(
        chunk_id="c001",
        raw_phrases=["concerns about artificial intelligence", "privacy risks"],
        retained_phrases=["privacy risks"],
        dropped_by_filter=["concerns about artificial intelligence"],
        sentinel_returned=False,
        error=None,
    )
    assert len(result.dropped_by_filter) == 1
    assert result.retained_phrases == ["privacy risks"]
```

**Checkpoint round-trip:**

```python
def test_checkpoint_roundtrip(tmp_path):
    data = {"key": [1, 2, 3]}
    du.save_checkpoint(data, tmp_path / "ckpt.pkl")
    loaded = du.load_checkpoint(tmp_path / "ckpt.pkl")
    assert loaded == data
```

**Sensitivity output paths** (from CIP-0002 — verify prefix logic without running KMeans):

```python
def test_sensitivity_output_prefix(tmp_path, monkeypatch):
    # Monkeypatch KMeans and LLM calls
    ...
    du.run_sensitivity(k=3, kind="concern", ..., output_folder=str(tmp_path))
    assert (tmp_path / "concern_sensitivity_stable_core_k3.csv").exists()
    assert not (tmp_path / "benefit_sensitivity_stable_core_k3.csv").exists()
```

Functions that require OpenAI API calls (`extract_phrases`, `label_cluster`, `get_embeddings_batch`) are tested with `unittest.mock.patch` or `pytest-mock`, not against the live API.

### Stripping stored notebook outputs

Install and configure `nbstripout` as a git filter so outputs are never committed:

```bash
pip install nbstripout
nbstripout --install  # installs git filter in .git/config
```

After this, `git add public_dialogue_analyser_v12b_4.ipynb` will automatically strip outputs before staging. The `outputs/` directory (CSV, HTML, PNG, JSON) remains the authoritative results record.

## Implementation Plan

1. **Audit all cells** — list every function definition and the cell(s) it appears in (use `jq` on the notebook JSON or a Python script)

2. **Create `dialogue_utils.py`** with the module structure above — copy the canonical version of each function; add `kind` parameter to `label_cluster` and the new `extract_phrases` unified function; add `run_sensitivity` (from CIP-0002)

3. **Create `tests/test_dialogue_utils.py`** — write tests for all pure utility functions first (no mocking needed), then add mocked tests for API-dependent functions

4. **Run `pytest tests/`** and confirm all tests pass against the freshly extracted module

5. **Update the notebook** — replace all inline function definitions with `import dialogue_utils as du` and calls to `du.<function>`. The notebook becomes orchestration only.

6. **Remove duplicate definitions** from Part I, Part II, Part IV — each cell now calls the module

7. **Install `nbstripout`** and verify the file size drops to a few hundred KB

8. **Update section numbering** and add Table of Contents markdown cell

9. **Run the full notebook** on a sample (or from checkpoints) and compare outputs to pre-refactor versions — `cluster_summary.csv` should be identical

## Backward Compatibility

- Output file schemas are unchanged; this CIP does not alter algorithm logic
- The `.ipynb` will no longer have stored outputs — anyone viewing outputs in the notebook interface must re-run or use `outputs/`
- Colab users need `dialogue_utils.py` available (via upload zip or Drive mount — SECTION 0 will document this)
- Section numbering changes are markdown-only; no code depends on section labels

## Testing Strategy

- `pytest tests/` must pass with 0 failures after Step 4
- After Step 9, `cluster_summary.csv` from a checkpoint run should be byte-identical to the pre-refactor version (or explain any expected differences)
- `git diff` on a one-line source change should show only that line, not megabytes of output JSON
