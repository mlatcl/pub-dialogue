# CIP-0002: Fix sensitivity analysis: deduplicate run_for_k and separate output prefixes

## Summary

Fix the sensitivity analysis (PART IV) so that concern and benefit runs use a single parameterised function, write to distinct output paths, and carry correct plot labels. This removes the risk that a full top-to-bottom run overwrites concern sensitivity outputs with benefit outputs. Implements [REQ-0002](../requirements/req0002_sensitivity-correctness.md). Depends on [CIP-0003](./cip0003.md) for overall code structure.

## Motivation

PART IV of the notebook runs sensitivity analysis at k ∈ {60, 75, 90} for both concerns (IV.A, cell 104) and benefits (IV.B, cell 114). The two sections define a function with the **same name** (`run_for_k`) and write outputs to the **same file paths** (e.g. `sensitivity_stable_core_k75.csv`). In a full run the benefit section executes last, so concern sensitivity files are overwritten by benefit runs.

Evidence of this problem:

- The `outputs/` directory contains `benefit_sensitivity_*` files with prefixes not written by any notebook cell — they must have been renamed manually after the run
- Cell 114's plot labels still read "Share of all extracted **concerns**" (copy-paste from IV.A that was never corrected)

This makes the sensitivity outputs in `outputs/` unreliable unless you know which manual corrections were applied, and makes the analysis non-reproducible.

## Detailed Description

### The bug in full

Part IV.A (cell 104) defines `run_for_k(k, embeddings_normalized, concerns_df, ...)` and writes:
- `sensitivity_stable_core_k{k}.csv`
- `sensitivity_fingerprint_k{k}.html`
- `sensitivity_time_metrics_k{k}.csv`
- `sensitivity_entropy_k{k}.png`
- `sensitivity_lens_k{k}.csv`
- `sensitivity_radar_k{k}.html`

Part IV.B (cell 114) redefines `run_for_k(k, ...)` binding to `benefits_df` and benefit embeddings, but writes to **the same set of paths**. The definition in cell 114 replaces the one from cell 104 in the kernel's namespace.

### Fix: single parameterised function

Replace both definitions with one function:

```python
def run_sensitivity(
    k: int,
    kind: str,                  # "concern" or "benefit"
    embeddings_normalized,      # the appropriate embedding matrix
    df,                         # concerns_df or benefits_df
    output_folder: str,
    # ... other params
):
    prefix = f"{kind}_sensitivity"
    # all file writes use f"{output_folder}/{prefix}_stable_core_k{k}.csv" etc.
    # plot labels use kind.capitalize() (e.g. "Concern" or "Benefit")
    ...
```

Call sites:

```python
# Part IV.A — concerns
for k in [60, 75, 90]:
    run_sensitivity(k, "concern", embeddings_normalized, concerns_df, OUTPUT_FOLDER)

# Part IV.B — benefits
for k in [60, 75, 90]:
    run_sensitivity(k, "benefit", benefits_embeddings_normalized, benefits_df, OUTPUT_FOLDER)
```

### Output path scheme

| Current (broken) | Fixed concern path | Fixed benefit path |
|---|---|---|
| `sensitivity_stable_core_k75.csv` | `concern_sensitivity_stable_core_k75.csv` | `benefit_sensitivity_stable_core_k75.csv` |
| `sensitivity_fingerprint_k75.html` | `concern_sensitivity_fingerprint_k75.html` | `benefit_sensitivity_fingerprint_k75.html` |
| `sensitivity_time_metrics_k75.csv` | `concern_sensitivity_time_metrics_k75.csv` | `benefit_sensitivity_time_metrics_k75.csv` |
| `sensitivity_entropy_k75.png` | `concern_sensitivity_entropy_k75.png` | `benefit_sensitivity_entropy_k75.png` |
| `sensitivity_lens_k75.csv` | `concern_sensitivity_lens_k75.csv` | `benefit_sensitivity_lens_k75.csv` |
| `sensitivity_radar_k75.html` | `concern_sensitivity_radar_k75.html` | `benefit_sensitivity_radar_k75.html` |

### Plot label fix

Inside `run_sensitivity`, all plot labels, axis titles, and HTML headings that currently contain hard-coded "concerns" should use the `kind` parameter:

```python
plt.title(f"Share of all extracted {kind}s by cluster (k={k})")
```

### Relationship to CIP-0003

[CIP-0003](./cip0003.md) addresses the broader code refactoring. This CIP targets the sensitivity bug specifically and can be implemented independently, but the refactored function should be placed in whichever shared utility cell CIP-0003 establishes.

## Implementation Plan

1. **Identify all parameters that differ** between the concern and benefit `run_for_k` implementations (embedding matrix, dataframe, column names, any per-track thresholds)

2. **Write `run_sensitivity(k, kind, embeddings, df, output_folder, ...)`** with parameterised paths and labels — a single definition, placed in the shared utility section established by CIP-0003

3. **Update Part IV.A** to call `run_sensitivity(k, "concern", ...)` — remove the old `run_for_k` definition from this cell

4. **Update Part IV.B** to call `run_sensitivity(k, "benefit", ...)` — remove the old `run_for_k` definition from this cell

5. **Rename existing output files** in `outputs/` to match the new scheme (or delete and re-run sensitivity from checkpoint)

6. **Run a full sensitivity pass** for both tracks and verify that 12 output files are produced (6 file types × 2 tracks at k=75; or 36 at k ∈ {60,75,90})

7. **Spot-check plot labels** in the HTML outputs to confirm they read "benefit" where expected

## Backward Compatibility

- Output file names change. Any downstream analysis or paper figure that references the old sensitivity filenames will need updating. The new names are strictly clearer, so this is a breaking-but-correct change.
- The function `run_for_k` will no longer exist in the kernel. If any other cell references it by name, those references must be updated to `run_sensitivity`.

## Testing Strategy

- After implementation, run the sensitivity section for both tracks
- Verify that the `outputs/` directory contains exactly the expected concern_sensitivity_* and benefit_sensitivity_* files
- Open one concern and one benefit HTML output and confirm plot labels are correct
- Confirm no `sensitivity_*` files (without the concern/benefit prefix) remain in `outputs/`
